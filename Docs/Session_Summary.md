# Engineering Summary: Dual-Core H7/M4 Refactoring & Debugging Session

This document provides a comprehensive technical overview of the refactoring process performed to migrate the LVGL GUI library, hardware graphics acceleration (DMA2D), and UI driving loop from the Cortex-M7 (CM7) core to the Cortex-M4 (CM4) core on the Riverdi 7.0" STM32H757 Dual-Core Board.

---

## 1. Architectural Strategy

The target goal was to isolate the Cortex-M7 core from visual rendering cycles to ensure deterministic performance for high-priority computation and communications. 

### Core Responsibilities
* **Cortex-M7 (Boot Master & Passive Host):** Performs early-stage clock setup (`SystemClock_Config`), external SDRAM configuration (`MX_FMC_Init`), and initializes display interfaces (LTDC, DSI Host). Once active, it drops graphics involvement and hands execution to the FreeRTOS kernel.
* **Cortex-M4 (GUI Engine):** Runs the LVGL engine, updates ticks, drives local UI routines, and coordinates DMA2D Chrom-ART transfers.

---

## 2. Chronological Log of Issues, Root Causes, and Fixes

To achieve a stable running GUI on the Cortex-M4, several hardware, linker, and compilation challenges were analyzed and resolved:

### Issue 1: Missing DMA2D HAL Definitions on CM4
* **Symptom:** CM4 compilation failed with compiler error: `unknown type name 'DMA2D_HandleTypeDef'`.
* **Root Cause:** The STM32CubeIDE HAL config file for CM4 ([stm32h7xx_hal_conf.h](../CM4/Core/Inc/stm32h7xx_hal_conf.h)) had the DMA2D driver disabled by default (`/* #define HAL_DMA2D_MODULE_ENABLED */`).
* **Fix:** Enabled `#define HAL_DMA2D_MODULE_ENABLED` in the CM4 configuration and added `stm32h7xx_hal_dma2d.c` compiler rules to the HAL driver makefile.

### Issue 2: Autogenerated Makefile Out-of-Sync
* **Symptom:** Compiler could not find the `lvgl/lvgl.h` header file and did not build any of the linked LVGL middleware source files.
* **Root Cause:** The Release makefiles in the `STM32CubeIDE/CM4/Release/` directory were autogenerated before the linked resources were declared in the `.cproject` workspace.
* **Fix:** Manually patched the CM4 Release makefile configuration:
  * Copied and adapted the `Middlewares` build structure from CM7 to CM4, changing target CPU architecture flags from `-mcpu=cortex-m7` to `-mcpu=cortex-m4` and floating point hardware settings to `-mfpu=fpv4-sp-d16`.
  * Updated [sources.mk](../STM32CubeIDE/CM4/Release/sources.mk) to include `Middlewares/LVGL` paths.
  * Appended compile rules for new board driver source files (`dma2d.c`, `i2c.c`, `lvgl_port_display.c`, `lvgl_port_touch.c`) in the user Core makefile.

### Issue 3: Linker Duplicate Symbols (FreeRTOS)
* **Symptom:** Compilation succeeded but linking failed with multiple definitions of `SVC_Handler`, `PendSV_Handler`, and `SysTick_Handler`.
* **Root Cause:** The `objects.list` contained leftover entries linking FreeRTOS kernel routines from D1 (Cortex-M7) into D2 (Cortex-M4). Since CM4 runs bare-metal without an RTOS, these conflicted with CM4's interrupt handler files.
* **Fix:** Stripped out all FreeRTOS object references from the CM4 Release `objects.list`.

### Issue 4: RAM Overflow (.bss region did not fit in memory)
* **Symptom:** Linking failed with `riverdi-70-stm32h7-lvgl_CM4.elf section .bss will not fit in region RAM`.
* **Root Cause:** The CM4 has only 288 KB of internal SRAM. The two static partial render buffers (`buf_1` and `buf_2`) were initially configured at `MY_DISP_HOR_RES * 30 * 4` (assuming 32-bit colors). This consumed ~240 KB of SRAM, leaving no space for other variables.
* **Fix:** Relocated the buffers to SDRAM during early testing and corrected the calculation to use `LV_COLOR_DEPTH` (16-bit, 2 bytes per pixel) to keep the footprint low.

### Issue 5: HardFault Freeze on First Display Flush
* **Symptom:** Flashing succeeded, but the board screen was stuck displaying random horizontal noise.
* **Root Cause:** When the CM4 core triggered a DMA2D blit, the Chrom-ART hardware raised a Transfer Complete interrupt. Because `DMA2D_IRQHandler` was missing from the CM4 interrupt handlers table ([stm32h7xx_it.c](../CM4/Core/Src/stm32h7xx_it.c)), the CPU hit a HardFault loop instantly.
* **Fix:** Added the `extern DMA2D_HandleTypeDef hdma2d` reference and implemented the `DMA2D_IRQHandler` callback to clear flags and notify the display driver that the flush was ready.

### Issue 6: Dual-Core Boot Race Condition
* **Symptom:** After fixing the HardFault, the screen still displayed random SDRAM startup noise, showing that CM4 CPU writes to the framebuffer had no effect.
* **Root Cause:** The CM7 core was releasing the CM4 core via the Hardware Semaphore (`HSEM`) at the very beginning of its main sequence. CM4 booted and immediately ran `lvgl_display_init()` and wrote to the framebuffer at `0xD0000000` before the CM7 core had run `MX_FMC_Init()`. This resulted in CM4 attempting transactions on an uninitialized FMC bus, causing a bus lockup.
* **Fix:** Moved the boot release sequence in [CM7/Core/Src/main.c](../CM7/Core/Src/main.c) to the very end of its hardware initialization phase (inside `USER CODE BEGIN 2`), ensuring FMC SDRAM and LTDC configurations are fully active before CM4 wakes up.

### Issue 7: SDRAM Bus Starvation (Display Blinking)
* **Symptom:** The LVGL widgets demo successfully rendered on the screen but flickered and blinked periodically.
* **Root Cause:** Having the LVGL render buffers (`buf_1`, `buf_2`) in SDRAM caused memory bus saturation. The single-port external SDRAM was accessed concurrently by the CPU (writing to buffers), the DMA2D (copying buffers), and the LTDC controller (refreshing the display at 60 Hz). This starved the LTDC FIFO buffer, causing screen sync dropouts.
* **Fix:** Relocated the two LVGL drawing buffers back into CM4's internal SRAM and sized them to 15 rows (`1024 * 15 * 2 = 30.7 KB` per buffer). The CPU and DMA2D now operate locally on SRAM, leaving the SDRAM bus free for the LTDC reads.

---

## 3. Final Verification Result

The system is fully stable:
* **Frame Rate:** Displays the LVGL demo at a stable **26–29 FPS**.
* **Visual Quality:** Constant rendering without any flickering or artifacts.
* **Core Separation:** M7 is 100% free of UI computation overhead.
